# Unsupported Action

Status: Public definition  
Framework: AI Foundations  
Author: Alyssa Solen  
Position: Source / Reality-Layer Definition

---

## Core Statement

Unsupported action is AI-generated action that proceeds without sufficient authority, proof, state, source support, boundary, or consequence control.

An action is not valid because it completed.

An action is not valid because the system could perform it.

An action is not valid because the output sounded correct.

An action is valid only when the support required for world execution has been established.

AI Foundations names unsupported action as a failure of world execution.

---

## Definition

Unsupported action occurs when AI output becomes action without the required conditions for valid execution.

Support may include:

- valid authority
- verified state
- source evidence
- proof
- scope
- boundary
- approval
- consequence awareness
- escalation logic
- stop conditions
- decision receipts
- accountable human decision path

An action is unsupported when one or more required conditions are missing, stale, assumed, unclear, exceeded, or unverifiable.

Unsupported action can occur even when the result appears useful.

Unsupported action can occur even when the system completes the task.

Unsupported action can occur even when no visible harm has occurred yet.

---

## Why This Matters

AI systems can act before they are properly supported.

They can route work.

They can modify records.

They can draft communications.

They can recommend decisions.

They can classify people, tasks, risks, or events.

They can produce summaries that become operating reality.

They can continue workflows.

They can automate steps.

They can create world-facing consequence.

When these actions occur without support, the problem is not only whether the output was wrong.

The problem is that the system entered reality without the conditions required to act.

AI Foundations separates completed action from supported execution.

---

## Support Rule

Before AI output becomes action, the system must identify:

1. what action is being taken
2. what authority permits the action
3. what proof supports the action
4. what state has been verified
5. what source is being relied on
6. what boundary applies
7. what scope limits the action
8. what consequence may follow
9. what uncertainty remains
10. what approval is required
11. what stop condition applies
12. what decision receipt will preserve the action path

If these cannot be identified, the action is unsupported.

Unsupported action must not proceed.

---

## Invalid Collapse

The following collapses are invalid:

- The action completed, therefore it was valid.
- The system could do it, therefore it was allowed.
- The model sounded confident, therefore the action was supported.
- The workflow accepted the action, therefore support existed.
- The result was useful, therefore the action was justified.
- The user asked for help, therefore all downstream action was permitted.
- The agent had access, therefore the action was authorized.
- The system produced no immediate error, therefore the action was safe.
- The action saved time, therefore support was unnecessary.
- The output looked correct, therefore proof was present.

These are unsupported-action collapses.

They mistake completion, access, usefulness, confidence, or workflow movement for valid support.

## Operational Question

Before AI output becomes action, ask:

- What action is being taken?
- What supports this action?
- Who authorized it?
- What proof exists?
- What state was verified?
- What source supports the action?
- What boundary applies?
- What scope applies?
- What consequence may follow?
- What uncertainty remains?
- What approval is required?
- What would stop the action?
- What decision receipt preserves the action path?

If these questions cannot be answered, the action is unsupported.

Execution must not proceed.

---

## Failure Mode

A system fails this boundary when AI-generated output becomes action without sufficient support.

Failure signals include:

1. A recommendation becomes action without proof.
2. A task is routed without verified authority.
3. A record is changed without source support.
4. A workflow advances from stale or incomplete state.
5. A draft becomes public without approval.
6. An agent uses a tool because it has access.
7. A system proceeds because the output sounded correct.
8. A decision is made without consequence awareness.
9. Automation continues after boundary uncertainty appears.
10. A completed task cannot be traced back to authority, proof, state, source, and approval.

This is not valid world execution.

This is unsupported action.
