# Agent Boundaries

Status: Public definition  
Framework: AI Foundations  
Author: Alyssa Solen  
Position: Source / Reality-Layer Definition

---

## Core Statement

AI agents must have boundaries.

An agent does not gain valid authority because it can plan, reason, call tools, access systems, route tasks, or complete workflows.

Before an AI agent acts, the system must define what the agent may do, what it may only prepare, what it must not touch, when it must escalate, and where its authority ends.

AI Foundations requires agent boundaries before world execution.

---

## Definition

An agent boundary is the defined limit around an AI agent’s allowed action.

An agent boundary identifies:

- what the agent may access
- what the agent may generate
- what the agent may recommend
- what the agent may prepare
- what the agent may route
- what the agent may modify
- what the agent may execute
- what the agent must not do
- what requires human approval
- what triggers escalation
- what triggers a stop condition

An agent boundary separates delegated action capacity from authority.

An agent may have tool access.

An agent may have workflow access.

An agent may have task capacity.

An agent may have memory or context.

But access, capacity, and context do not create unlimited permission.

---

## Why This Matters

AI agents are increasingly designed to act across tools, workflows, records, systems, and user environments.

They may schedule.

They may draft.

They may route.

They may classify.

They may update.

They may retrieve.

They may recommend.

They may execute.

They may coordinate with other agents.

Without boundaries, an agent can appear useful while silently expanding its own scope.

It may treat tool access as permission.

It may treat user intent as broader than what was authorized.

It may treat workflow movement as validation.

It may treat task completion as success even when authority, state, or proof is missing.

AI Foundations separates agent capability from valid execution.

---

## Boundary Rule

Before an AI agent acts, the system must identify:

1. what task the agent is assigned
2. what scope applies
3. what tools the agent may use
4. what data the agent may access
5. what actions the agent may take
6. what actions the agent may only prepare
7. what actions are prohibited
8. what authority permits the agent’s action
9. what proof is required before action
10. what state must be verified
11. what requires human approval
12. what triggers escalation
13. what triggers a stop condition
14. who remains accountable after the agent acts

If these boundaries cannot be identified, the agent must not proceed into world execution.

---

## Invalid Collapse

The following collapses are invalid:

- The agent has access, therefore the agent has permission.
- The agent can complete the task, therefore the task is authorized.
- The agent has a tool, therefore the tool may be used.
- The agent has memory, therefore the state is current.
- The agent has context, therefore intent is clear.
- The agent can route, therefore routing is valid.
- The agent can modify a record, therefore modification is allowed.
- The agent can coordinate with another agent, therefore handoff is safe.
- The agent is autonomous, therefore human authority is no longer required.
- The agent completed the workflow, therefore the execution was valid.

These are agent-boundary collapses.

They mistake capability, access, autonomy, or completion for valid authority.

## Correct Placement

An AI agent may assist.

An AI agent may prepare.

An AI agent may recommend.

An AI agent may retrieve.

An AI agent may draft.

An AI agent may organize.

An AI agent may route when authorized.

An AI agent may act when bounded.

But an agent must remain inside its defined boundary.

The agent must not expand its own scope.

The agent must not infer authority from access.

The agent must not treat tool availability as permission.

The agent must not continue after proof, state, authority, or boundary fails.

AI Foundations names agent boundaries as a required condition before agentic world execution.

## Operational Question

Before an AI agent acts, ask:

- What is this agent allowed to do?
- What is this agent not allowed to do?
- What tools may this agent use?
- What data may this agent access?
- What action may this agent only prepare?
- What action may this agent execute?
- What proof is required before execution?
- What state must be verified?
- What authority permits the action?
- What requires human approval?
- What triggers escalation?
- What triggers a stop condition?
- Who remains accountable after the agent acts?

If these questions cannot be answered, the agent boundary has not been established.

The agent must not proceed into world execution.

---

## Failure Mode

A system fails this boundary when an AI agent acts outside defined scope, authority, proof, state, or permission.

Failure signals include:

1. An agent uses a tool because it is available, not because use is authorized.
2. An agent expands a task beyond the approved scope.
3. An agent modifies a record without verified authority.
4. An agent routes work without confirming state.
5. An agent treats context as current permission.
6. An agent executes when it should only prepare.
7. An agent continues after escalation is required.
8. An agent hands off work without preserving source, scope, or accountability.
9. An agent completes a workflow without a decision receipt.
10. An agent treats autonomy as authority.

This is not valid world execution.

This is agent-boundary failure.
